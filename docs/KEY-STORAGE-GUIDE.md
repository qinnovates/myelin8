# Key Storage Guide — Securing Your PQKC Encryption Key

This guide covers **where** to store the private key generated by `age-keygen`
for the engram's optional post-quantum encryption feature.

**You are responsible for your own key management.** This engine never stores,
transmits, or manages your private key. It only uses it locally for
encrypt/decrypt operations.

---

## Quick Context: What Key Are We Protecting?

When you run `age-keygen`, you get:
- **Public key** (`age1...`) — safe to share, goes in your config
- **Private key** (`AGE-SECRET-KEY-...`) — this is the secret. Lose it = lose access to cold-tier data

age v1.3.0+ uses **ML-KEM-768 hybrid with X25519** (NIST FIPS 203 compliant).
This means your encrypted data is protected against both classical and quantum attacks.

## Storage Options (Ranked by Security)

### Tier 1: Hardware-Backed (Strongest)

#### Cloud KMS (AWS KMS / Azure Key Vault / GCP KMS)

Wrap your age private key with a KMS-managed key. The age key is stored
encrypted; decryption requires an authenticated API call to the KMS.

**Pros:** HSM-backed (FIPS 140-2 Level 2+), automatic rotation, audit logs
**Cons:** Vendor lock-in, requires cloud account, API latency on every recall
**Risk:** IAM misconfiguration is the #1 real-world failure mode. Principle of least privilege is critical.

| Service | Docs |
|---------|------|
| AWS KMS | https://docs.aws.amazon.com/kms/latest/developerguide/ |
| Azure Key Vault | https://learn.microsoft.com/en-us/azure/key-vault/ |
| GCP KMS | https://cloud.google.com/kms/docs |

**Setup pattern:**
```bash
# Encrypt your age key with KMS before storing
aws kms encrypt --key-id alias/engram \
  --plaintext fileb://key.txt \
  --output text --query CiphertextBlob > key.txt.encrypted

# Decrypt on demand (requires IAM auth)
# WARNING: Avoid /tmp (world-readable). Use mktemp with restricted perms.
KEYFILE=$(mktemp) && chmod 600 "$KEYFILE"
aws kms decrypt --ciphertext-blob fileb://key.txt.encrypted \
  --output text --query Plaintext | base64 -d > "$KEYFILE"
# Use $KEYFILE as identity_path, then shred it
shred -u "$KEYFILE"
```

### Tier 2: Enterprise Secrets Management

#### HashiCorp Vault

Store the age private key as a Vault secret. Retrieve it programmatically
when engram needs to decrypt.

**Pros:** Fine-grained ACL, audit logging, dynamic secrets, transit engine can do encryption-as-a-service
**Cons:** High setup complexity (unsealing, HA, storage backend), Enterprise features cost money
**Risk:** Vault starts sealed on every restart. If unseal keys are lost, ALL data is permanently inaccessible. Root token must be revoked after initial setup.

| Resource | URL |
|----------|-----|
| Vault Docs | https://developer.hashicorp.com/vault/docs |
| KV Secrets Engine | https://developer.hashicorp.com/vault/docs/secrets/kv/kv-v2 |
| PQ Support (Enterprise) | https://www.hashicorp.com/en/blog/nist-s-post-quantum-cryptography-standards-our-plans |

**Setup pattern:**
```bash
# Store key in Vault
vault kv put secret/engram/age-key \
  private_key=@key.txt

# Retrieve on demand — avoid /tmp, use mktemp
KEYFILE=$(mktemp) && chmod 600 "$KEYFILE"
vault kv get -field=private_key secret/engram/age-key > "$KEYFILE"
# Use, then shred
shred -u "$KEYFILE"

# Even better: pipe directly, never touch disk
vault kv get -field=private_key secret/engram/age-key \
  | age -d -i /dev/stdin -o decrypted.txt encrypted.txt.age
```

### Tier 3: Kubernetes Secrets (If You're Already in K8s)

#### Kubernetes Secrets + KMS Provider

Store the age key as a K8s secret, BUT only if you've enabled encryption
at rest with a KMS provider.

**Pros:** Native K8s integration, works with existing RBAC
**Cons:** Default K8s secrets are base64-encoded, NOT encrypted. etcd backups leak everything.
**Risk:** Most clusters ship with NO encryption at rest. base64 is encoding, not security. You MUST enable `--encryption-provider-config` with a real provider.

| Resource | URL |
|----------|-----|
| K8s Secrets Docs | https://kubernetes.io/docs/concepts/configuration/secret/ |
| Encrypting Data at Rest | https://kubernetes.io/docs/tasks/administer-cluster/encrypt-data/ |
| Good Practices | https://kubernetes.io/docs/concepts/security/secrets-good-practices/ |

**Minimum safe setup:**
1. Enable etcd encryption with `aesgcm` or `kms` provider
2. Restrict RBAC: only the service account running engram gets `get` on the secret
3. Never grant `list` on secrets namespace-wide (returns ALL values, not just names)
4. Encrypt etcd backups separately

**WARNING:** Never commit this manifest with a real key value. Store the key via `kubectl create secret` instead.

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: engram-age-key
  namespace: ai-tools
type: Opaque
stringData:
  # NEVER commit this file with a real key value
  # Use: kubectl create secret generic engram-age-key \
  #   --from-file=key.txt=./key.txt -n ai-tools
  key.txt: "<YOUR-AGE-SECRET-KEY-HERE>"
```

### Tier 4: Password Manager (Individual Developer)

#### 1Password SSH Agent

Store the age private key in your 1Password vault. 1Password can act as
an SSH agent, but for age keys you'd retrieve via CLI.

**Pros:** AES-256-GCM encryption, Master Password + Secret Key (128-bit), SSH agent integration
**Cons:** Single point of failure (master password compromise), not for CI/CD or automation, no PQ key generation
**Risk:** Biometric unlock weakens security to biometric/PIN strength. Cloud-synced vault has larger attack surface than air-gapped. Not suited for non-interactive use.

| Resource | URL |
|----------|-----|
| 1Password SSH | https://developer.1password.com/docs/ssh/ |
| 1Password CLI | https://developer.1password.com/docs/cli/ |

**Setup pattern:**
```bash
# Store in 1Password
op item create --category="Secure Note" \
  --title="engram age key" \
  --vault="Private" \
  "notesPlain=$(cat key.txt)"

# Retrieve on demand — avoid /tmp, use mktemp
KEYFILE=$(mktemp) && chmod 600 "$KEYFILE"
op item get "engram age key" --fields notesPlain > "$KEYFILE"
# Use, then shred
shred -u "$KEYFILE"
```

#### Bitwarden

Similar to 1Password. Open-source vault. AES-256 with PBKDF2 key derivation.

| Resource | URL |
|----------|-----|
| Bitwarden CLI | https://bitwarden.com/help/cli/ |
| Bitwarden SSH Agent | https://bitwarden.com/help/ssh-agent/ |

**Risk:** Same single-point-of-failure concerns as 1Password. Master password is the weakest link.

### Tier 5: File-Based (Simplest, Least Secure)

#### age Passphrase Protection

Encrypt the key file itself with a passphrase. Requires typing the passphrase
on every recall operation.

```bash
# Protect the key with a passphrase
age -p -o key.txt.age key.txt
shred -u key.txt   # destroy the plaintext

# To use: decrypt key first
age -d key.txt.age > /tmp/key.txt
# Run engram recall, then shred
shred -u /tmp/key.txt
```

**Risk:** Passphrase strength is the entire security boundary. No audit log. No rotation. No recovery if forgotten.

#### macOS Keychain (Local Only)

```bash
# Store in Keychain
security add-generic-password -s "engram" -a "age-key" \
  -w "$(cat key.txt)" -T ""

# Retrieve
security find-generic-password -s "engram" -a "age-key" -w > /tmp/key.txt
```

**Risk:** Tied to macOS login password. If someone has your Mac password, they have the key. Not portable across machines.

## Risk Comparison Matrix

| Option | Encryption | Key Never on Disk | Audit Log | PQ Ready | Automation | Setup Effort |
|--------|-----------|-------------------|-----------|----------|------------|-------------|
| Cloud KMS | HSM-backed | Yes (HSM) | Yes | No* | Yes | Medium |
| HashiCorp Vault | Transit engine | Yes (in Vault) | Yes | Enterprise | Yes | High |
| K8s + KMS | KMS-backed | Depends | Limited | No* | Yes | Medium |
| 1Password | AES-256-GCM | In vault | No | No | Limited | Low |
| Bitwarden | AES-256 | In vault | No | No | Limited | Low |
| Passphrase | age | While encrypted | No | Yes (age) | No | Lowest |
| macOS Keychain | Keychain | In Keychain | No | No | macOS only | Low |

*Cloud KMS PQ support not yet announced by any provider. Your data is still safe because
AES-256 (symmetric) is not meaningfully weakened by quantum computers (Grover's algorithm
reduces it to ~128-bit equivalent, which remains secure).

## What NOT To Do

1. **Never commit the private key to git.** Add `key.txt` and `*.age` to `.gitignore`.
2. **Never store the key in environment variables** in CI/CD logs or Dockerfiles.
3. **Never email the key.** Use a secure channel (1Password share, Vault policy, KMS grant).
4. **Never use the same key across unrelated systems.** Generate one key per use case.
5. **Never skip `shred -u`** after using a temporary plaintext key file. `rm` doesn't overwrite disk sectors.

---

## Key Rotation

When rotating keys:
1. Generate new keypair: `age-keygen -o key-new.txt`
2. Decrypt all cold/warm artifacts with old key
3. Update config with new public key
4. Re-run tiering engine (will re-encrypt with new key)
5. Securely destroy old key after verifying all artifacts re-encrypted
6. Update your vault with the new key
